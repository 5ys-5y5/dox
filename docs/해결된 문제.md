지금까지 해결한 문제는 아래와 같습니다.

1. 1회 클릭 `connected` 와 2회 클릭 `isolated` 선택 상태가 실제 drag 범위와 다르게 동작하던 문제
   - 재현: 같은 edge 를 1회 클릭한 뒤 drag 하면 connected cohort 가 아니라 단일 edge 만 움직이거나, 2회 클릭 뒤 drag 해도 connected cohort 전체가 움직였다.
   - 원인: 클릭 선택 상태와 실제 drag mutation target 계산이 분리되지 않아, 시각 선택 상태와 실제 이동 대상이 서로 다른 기준을 사용했다.
   - 해결: 선택 UI 상태와 drag mutation target 계산을 분리하고, drag 시작 시점에 별도의 `dragSelectionState` 를 확정해서 사용했다.
   - 결과: 1회 클릭은 connected cohort 기준으로, 2회 클릭은 isolated 기준으로 실제 drag 범위가 일치한다.

2. 2회 클릭 `isolated` 인데도 connected cohort 전체가 같이 움직이던 문제
   - 재현: edge 를 2회 클릭해 단일 선택 상태로 만든 뒤 drag 하면, 선택한 edge 가 아니라 같은 선상의 connected edge 들이 전부 같이 움직였다.
   - 원인: `selected_edge_clicked` 와 peer follower 를 같은 집합으로 취급해 isolated selection 이 mutation 단계에서 connected selection 으로 확장됐다.
   - 해결: resize intent 계약에서 `selected_edge_clicked` 단독 selection 과 실제 follower(peer, passive follower) 를 구분했다.
   - 결과: 2회 클릭 isolated 상태에서는 클릭한 edge 만 직접 선택되고, 필요한 peer follower 만 수동 추종한다.

3. 연관 없는 엣지가 같이 움직이거나 blocker 로 작동하던 문제
   - 재현: `band-3-cell-1:right` 같은 edge 를 drag 하면 unrelated edge 가 같이 움직이거나, 이동 자체를 막았다.
   - 원인: drag 대상 계산에 unrelated edge 와 generic sibling snap blocker 가 섞여 있었다.
   - 해결: drag 대상 계산을 active role 집합(`selected_edge_clicked`, `selected_edge_auto_multi`, `peer_edge`)으로 제한하고 unrelated edge 를 제외했다.
   - 결과: 선택되지 않은 unrelated edge 는 직접 이동되지 않고, blocker 로도 작동하지 않는다.

4. drag 거리의 `n배` 만큼 이동하던 문제
   - 재현: edge 를 4px drag 했는데 실제로는 8px, 12px 처럼 중복 이동했다.
   - 원인: 같은 physical boundary 에 대한 width/height instruction 이 여러 target 에 중복 포함돼 같은 delta 가 여러 번 적용됐다.
   - 해결: 같은 physical boundary instruction 을 dedupe 하고, peer instruction 은 constraint 계산에만 참여시키고 실제 적용은 1회만 수행하게 했다.
   - 결과: drag 거리와 실제 이동 거리가 1:1 로 일치한다.

5. 특정 vertical chain 이동 후 1px 정렬 어긋남이 누적되던 문제
   - 재현: 같은 right-chain 을 반복해서 좌우 drag 하면 같은 수직선이 조금씩 벌어졌다.
   - 원인: pointerup 이후 vertical cohort 와 physical peer 가 live 좌표 기준으로 다시 완전히 수렴되지 않았다.
   - 해결: pointerup 뒤 vertical cohort/physical peer 재정렬과 live correction pass 를 넣어 exact line 으로 다시 맞췄다.
   - 결과: 왕복 drag 후에도 vertical chain 의 정렬 오차 누적이 크게 줄었다.

6. `status-history-1:left`, `band-3-cell-2:right`, `band-4-cell-2:right` 가 다른 row line 근처에서 최소 폭으로 접히던 문제
   - 재현: 위 edge 들을 다른 row 의 vertical line 근처로 drag 하면 `band-3-cell-2`, `band-4-cell-2` 가 최소 폭으로 갑자기 접혔다.
   - 원인: active vertical cluster 가 현재 drag baseline 대신 외부 same-side line 해석으로 갈아타면서 collapse 가 발생했다.
   - 해결: active vertical cluster 를 `baseline + applied delta` 로 잠그는 stabilize 보정을 넣었다.
   - 결과: 다른 row line 근처를 지나가도 active cluster 는 현재 drag 기준선을 유지하고 minimum width collapse 가 나지 않는다.

7. 다른 행의 시작점/끝점에 닿을 때 강제로 snap 되던 문제
   - 재현: edge 를 drag 하다가 실제로는 관계없는 다른 행의 endpoint 에 닿으면 갑자기 그 line 으로 튀었다.
   - 원인: autosnap 후보 계산에 endpoint-touch candidate 가 포함되어 있었다.
   - 해결: endpoint-touch candidate 를 autosnap 후보에서 제외했다.
   - 결과: 다른 행의 시작점/끝점만 맞닿는 경우에는 강제 snap 이 발생하지 않는다.

8. autosnap 이 작은 move 누적에서는 빠지던 문제
   - 재현: 한 번에 크게 drag 하면 snap 이 되지만, 작은 move 를 여러 번 누적해서 접근하면 snap 이 빠졌다.
   - 원인: autosnap 후보 계산이 현재 move delta 만 보고 누적 적용 delta 를 반영하지 않았다.
   - 해결: `currentAppliedDelta + requestedDelta` 기준으로 후보를 계산하도록 바꿨다.
   - 결과: 작은 move 가 누적되어 5px 미만 접근 상태가 되더라도 autosnap 이 정상 작동한다.

9. 같은 축이지만 span 이 겹치지 않는 edge 에는 autosnap 이 안 되던 문제
   - 재현: 같은 수직선/수평선상에 있지만 span 이 떨어져 있는 edge 근처로 drag 해도 autosnap 이 되지 않았다.
   - 원인: autosnap 후보가 `span overlap` 조건으로 잘려 있었다.
   - 해결: span-overlap 필터를 제거해 같은 수직선/수평선이면 candidate 가 되게 했다.
   - 결과: span 이 겹치지 않아도 같은 축에 있는 edge 에는 autosnap 이 적용된다.

10. autosnap point 가 여러 개 겹쳐 보여 엉뚱한 line 에 붙던 문제
   - 재현: same-side edge 와 opposite-side edge 후보가 동시에 있으면 사용자가 원하는 인접 line 이 아니라 다른 후보에 붙었다.
   - 원인: autosnap 후보 tie-break 가 side/span/endpoint 우선순위를 명확히 가지지 않았다.
   - 해결: `same-side 우선 -> span 관계 -> adjustment -> endpoint gap` 순서로 후보를 고르게 바꿨다.
   - 결과: 사용자가 의도한 same-side 인접 line 으로 우선 snap 된다.

11. 클릭만 해도 div 박스가 최소 크기로 붕괴하던 문제
   - 재현: 박스를 클릭하거나 selection 상태를 바꾸는 것만으로 전체 frame 이 최소 크기처럼 접혀 보였다.
   - 원인: normalization 시점이 너무 빨랐고, split shell rect 를 raw cell rect 에 의존해 불안정한 시점의 `12px` 최소값이 굳었다.
   - 해결: normalization 시점을 늦추고, shell rect 를 raw cell rect 대신 table boundary 누적합으로 계산했다.
   - 결과: 클릭/선택만으로 frame 이 최소 크기로 붕괴하지 않는다.

12. 편집 화면의 윤곽선이 extract 결과보다 두껍거나 끊겨 보이던 문제
   - 재현: `/templates/edit` 에서는 `/templates/extract` 결과와 달리 어떤 선은 두껍고 어떤 선은 끊겨 보였다.
   - 원인: native table border 와 editor overlay border 가 겹쳐서 두께와 seam 이 중복 표현됐다.
   - 해결: native border 를 숨기고 `page-inner` 기준 outline overlay 하나로 정리했다.
   - 결과: 편집 화면 윤곽선이 extract 결과와 가깝게 균일하게 보인다.

13. 외곽 윤곽선이나 dashed outline 이 사라지거나 이중 출력되던 문제
   - 재현: 어떤 box 는 외곽선이 사라지고, `band-0-header` 같은 dashed 항목은 이중 점선으로 보였다.
   - 원인: shell overlay 와 native dashed border 가 동시에 보이거나, perimeter 판정이 잘못되어 일부 side 가 빠졌다.
   - 해결: shell 자체 overlay 로 4면을 직접 그리고, dashed 는 native border 를 숨긴 뒤 overlay 만 남겼다.
   - 결과: 외곽선은 빠지지 않고, dashed outline 도 single-source 로만 보인다.

14. edge 선택이 안 되거나 특정 header box 에서 edge anchor 를 못 찾던 문제
   - 재현: `band-0-header`, `band-1-header` 같은 상단 header box 는 edge 클릭이 되지 않거나 anchor 를 잘못 잡았다.
   - 원인: normalized shell 과 raw frame node 가 다른 구조일 때 `frameGroupId` fallback 이 충분하지 않았다.
   - 해결: shell 과 raw frame node 양쪽에서 `frameGroupId` 를 fallback 하도록 바꿨다.
   - 결과: header box 를 포함한 모든 frame group 에서 edge selection 이 정상 작동한다.

15. `Shift + drag` 박스 선택 ghost 가 안 보이던 문제
   - 재현: `Shift + drag` selection 은 작동하지만 영역 표시 ghost 가 drag 중 바로 사라졌다.
   - 원인: marquee ghost 가 일반 `editor-ui` 제거 대상으로 같이 지워졌다.
   - 해결: ghost 를 일반 editor-ui 제거 대상에서 분리하고, 저장/추출 시에만 class 기준으로 제거하도록 바꿨다.
   - 결과: drag 중에는 selection rect 가 보이고, pointerup 뒤에는 정상 제거된다.

16. 박스 생성 drag 가 안 되거나 생성 후 mode 가 남아있던 문제
   - 재현: create mode 에서 drag 해도 박스가 만들어지지 않거나, 한 개 생성 후에도 create mode 가 계속 켜져 있었다.
   - 원인: create lifecycle 과 selection rerender 경로가 분리되어 있었고, 생성 후 mode 종료가 보장되지 않았다.
   - 해결: create lifecycle 을 다시 묶고, 1개 생성 후 create mode 를 자동 종료하도록 정리했다.
   - 결과: drag 로 박스가 생성되고, 생성 직후 create mode 가 정상 종료된다.

17. selected box/edge 표시가 과하게 겹쳐서 윤곽선을 가리던 문제
   - 재현: 선택한 box 나 edge 위에 badge, halo, fill 이 겹쳐 실제 outline 이 가려졌다.
   - 원인: 선택 표시와 실제 outline 이 같은 시각 레이어를 공유했다.
   - 해결: 선택 box 는 내부 `5px` inset fill 만, edge 는 선택된 side 의 `3px` inset line 만 그리도록 바꿨다.
   - 결과: 선택 상태는 보이지만 outline 자체는 가리지 않는다.

18. 텍스트 크기가 box resize 에 따라 같이 변하던 문제
   - 재현: box 높이나 너비를 조정하면 텍스트 font-size 나 line-height 도 같이 변했다.
   - 원인: box resize 뒤 text fit 경로가 `font-size`, `line-height` 를 다시 계산했다.
   - 해결: box 크기 변경이 `font-size`, `line-height` 를 다시 계산하지 않도록 끊었다.
   - 결과: box 외곽선/크기 변화와 텍스트 스타일이 독립적으로 유지된다.

19. `relative / absolute` 위치 관계가 모호해 resize 전파가 예측 불가능하던 문제
   - 재현: 어떤 box 는 아래 box 를 밀고, 어떤 box 는 밀지 않는데 기준이 코드상 분명하지 않았다.
   - 원인: anchor 개념과 flow 개념이 섞여 있었고, 위치 모드 계약이 명확하지 않았다.
   - 해결: 현재 구현 기준을 `relative = 아래 non-absolute 박스를 민다`, `absolute = 전파 제외` 로 단순화했다.
   - 결과: 위치 전파 규칙이 직관적이고 예측 가능해졌다.

20. autosnap 회귀
   - 재현: 안정화 보정을 넣은 뒤 autosnap 이 아예 약해지거나 사라졌다.
   - 원인: baseline lock 보정이 autosnap 의 마지막 미세 correction 까지 같이 눌렀다.
   - 해결: baseline lock 은 유지하고, live autosnap correction 을 한 번 더 수행하는 방식으로 복구했다.
   - 결과: collapse fix 를 유지한 채 autosnap 이 다시 작동한다.

21. `band-5-cell-3:left` 가 `status-history-1:left` 또는 `band-4-cell-2:right` 로 autosnap 될 때 `band-5-cell-2:right / band-5-cell-3:left` exact physical peer 경계가 분리되고, `band-5-cell-2` 박스가 최소 너비 지점까지 강제 이동되며 표가 파괴되던 문제
   - 재현: `selected_edge_clicked: band-5-cell-3:left` 를 오른쪽으로 drag 해 `status-history-1:left` 또는 `band-4-cell-2:right` 근처 autosnap 을 유도하면, local peer pair 가 분리되고 `band-5-cell-2` 가 접혔다.
   - 원인: `selected_edge_clicked + peer_edge` 만 있는 local exact-boundary width drag 에도 multi-row vertical correction / finalize 경로를 동일하게 태우고 있어서, local peer 쌍이 다른 row line 기준으로 과보정되었다.
   - 해결: simple exact physical boundary vertical drag 를 별도 판정(`isSimpleExactPhysicalBoundaryVerticalDrag`)하여, 이 경우에는 `stabilize / realign / live autosnap correction / drag-direction normalize / finalize` 같은 global correction 경로를 건너뛰고 local peer 상태만 유지하도록 분리했다.
   - 결과: `band-5-cell-3:left` 드래그 중 `band-5-cell-2:right / band-5-cell-3:left` 는 끝까지 같은 선을 유지하고, `status-history-1:left` 근처 autosnap 에서도 `band-5-cell-2` 가 최소 너비로 강제 수축되지 않는다.

22. `status-history-1:left` 를 `band-5-cell-2:right` 와 `band-5-cell-6:left` 사이에서 autosnap 되게 좌우 반복 이동하면 `status-history-1:right` 가 right-chain 에서 조금씩 오른쪽으로 밀려나던 문제
   - 재현: `selected_edge_clicked: status-history-1:left` 를 선택한 뒤 `band-5-cell-2:right` 로 snap, 다시 `band-5-cell-6:left` 로 snap 하는 왕복 drag 를 반복하면 `status-history-1:right` 가 cycle 당 `0.15625px` 수준으로 drift 누적되었다.
   - 원인: `applyOuterLeftWidthDelta(...)` 가 left edge 이동 후 table column width 합으로 shell width 를 다시 계산하면서 outer-right 를 정확히 보존하지 못했다. 그 결과 left snap 과 right snap 왕복을 반복할수록 `status-history-1:right` 가 오른쪽으로 누적 이동했다.
   - 해결: `applyOuterLeftWidthDelta(...)` 에서 left edge 이동 후 shell/table width sync 를 마친 다음, drag 시작 전 outer-right 와 현재 outer-right 차이를 다시 계산해 shell outer-right 가 정확히 원위치로 유지되도록 width 를 즉시 보정했다.
   - 결과: 같은 왕복 autosnap 경로를 5회 반복해도 `status-history-1:right` 와 right-chain(`band-5-cell-6:right` ~ `band-18-cell-2:right`)은 같은 좌표를 유지한다.

23. `메타데이터` 탭에서 div 박스를 추가 선택할 때마다 캔버스가 반짝이며 선택 표시와 `Role / Box Kind` 가시화 UI 가 사라지던 문제
   - 재현: `/templates/edit?templateId=5290e1f0-a4da-4a66-b180-7126e092b4fb` 에서 `메타데이터` 탭을 연 뒤 div 박스를 1개 선택하면 잠깐 정상처럼 보이지만, 추가 선택을 하거나 `Shift + drag` / `Shift + click` 으로 선택 집합을 바꾸는 순간 `박스 편집 캔버스` 가 꺼졌다 켜지듯 반짝였고, selection outline 과 `TEXT / SIGN` badge 가 함께 사라졌다가 나중에야 다시 보였다.
   - 원인: 선택 상태가 바뀔 때 부모 컴포넌트가 다시 렌더되면서 `dangerouslySetInnerHTML` 기반 preview surface 가 raw HTML 상태로 한 번 되돌아갔다. 브라우저에서 실제 측정하면 selection 추가 직후 `htmlLen` 이 editor-ui 포함 상태에서 raw 상태로 떨어졌다가 다음 프레임에 다시 복구되었고, 이 사이에 selection / metadata 시각화 overlay 가 모두 제거되었다. 여기에 클릭 핸들러가 state 의존성을 잡고 있어 선택 변경마다 캔버스 재주입이 더 자주 일어났다.
   - 해결: `TemplateEditWorkspace.tsx` 에서 preview surface 를 `TemplateEditPreviewSurface` 로 분리해 `React.memo` 로 고정하고, 선택/클릭/드래그 포인터 핸들러가 state 대신 ref(`selectedFrameGroupIdsRef`, `edgeRoleDiagnosticsRef`)를 기준으로 동작하도록 정리했다. 동시에 메타데이터 탭 전환 시 live selection UI 와 visual hint 를 동기 재적용하는 경로를 유지해 raw fallback 없이 바로 overlay 상태를 보존하게 했다.
   - 결과: 선택 추가/변경 시 캔버스가 raw HTML 로 되돌아가는 반짝임이 사라졌고, 단일 선택과 다중 선택 모두에서 selection outline, 선택 순서 badge, `Role / Box Kind` 가시화 UI 가 계속 유지된다.

24. `status-history-1` 부터 `band-18-cell-2` 까지 이어지는 connected frame cluster 의 윤곽선이 선택 후 사라져 보이던 문제
   - 재현: `/templates/edit?templateId=5290e1f0-a4da-4a66-b180-7126e092b4fb` 에서 초기 진입 직후에는 connected cluster 의 외곽선이 보이지만, div 박스를 1개 선택하거나 다중 선택을 추가하는 순간 윤곽선이 보이지 않게 되고, 선택을 더 추가할 때만 다시 잠깐 나타나는 것처럼 보였다.
   - 원인: connected cluster perimeter 계산 자체보다, 23번과 같은 preview surface 재주입이 직접 원인이었다. 선택 변경 직후 캔버스가 raw preview DOM 으로 한 번 갈리면서 `data-v106-frame-cluster-outline-overlay` 와 frame outline overlay 가 모두 제거되었고, 다음 프레임에서 다시 붙기 전까지는 connected cluster 의 최외곽선이 없는 상태로 보였다.
   - 해결: preview surface 를 memoized component 로 분리하고, selection 변경이 캔버스 재주입으로 이어지지 않도록 ref 기반 interaction 경로로 바꿨다. 그 결과 기존 connected cluster outline overlay 가 선택 전후 같은 live DOM 위에서 유지되며, selection UI 재적용 시에도 함께 남도록 정리했다.
   - 결과: `status-history-1` 부터 `band-18-cell-2` 까지 이어지는 connected frame cluster 의 최외곽선이 초기 상태뿐 아니라 단일 선택, 다중 선택, 메타데이터 탭 전환 뒤에도 계속 보인다.

25. `현재 템플릿 저장`을 누르면 정상 템플릿이 최소 크기 frame 들로 파괴되어 저장되던 문제
   - 재현: `/templates/edit` 에서 정상적으로 보이는 템플릿을 저장하기 전에는 `docs/html.html` 수준으로 giant band 폭·높이가 유지되지만, `현재 템플릿 저장` 뒤 저장본을 다시 열면 `docs/html-2.html`처럼 `band-3-cell-1`, `band-5-cell-6`, `status-history-1` 등의 frame 이 `12px` 단위 최소 크기로 줄어든 상태로 저장되었다.
   - 원인: 저장 직전 `extractEditorHtml(...)` 가 normalized preview shell 을 raw giant band 로 되돌릴 때 `denormalizePreviewFrameBands(...)` 를 사용한다. 이 과정에서 detached clone 안의 `col` / `tr` 크기를 `readTableColWidths(...)`, `readTableRowHeights(...)` 가 computed style 기준으로만 읽고 있었고, detached DOM 에서는 이 값이 비어 최소 fallback 값(`12px`)으로 떨어졌다.
   - 해결: `TemplateEditWorkspace.tsx` 에서 `readTableColWidths(...)`, `readTableRowHeights(...)` 가 computed style 보다 먼저 inline `style.width`, `style.height` 와 `width`, `height` attribute 를 읽도록 보강했다. 그래서 저장 직전 denormalize 단계에서도 실제 giant band 폭·높이를 그대로 복원해 `draftHtml` 로 저장하게 정리했다.
   - 결과: `현재 템플릿 저장` 이후에도 frame 폭·높이가 `docs/html.html` 수준으로 유지되고, 다시 불러오거나 새로고침해도 `12px` 최소 크기 frame 붕괴가 발생하지 않는다.

26. `선택 항목 저장`과 `현재 템플릿 저장`을 모두 거쳐도 새로고침 후 메타데이터/스타일 변경이 사라지던 문제
   - 재현: `/templates/edit` 에서 `boxKind`, `role`, `parentGroupId`, `runtimeMode` 같은 메타데이터나 `배경 색`, `모서리 반경` 같은 스타일을 변경한 뒤 `선택 항목 저장`과 `현재 템플릿 저장`을 순서대로 실행하면 화면에서는 반영된 것처럼 보였지만, 페이지를 새로고침하거나 다시 불러오면 변경사항이 모두 원래 상태로 돌아갔다.
   - 원인: 편집 화면의 live selection anchor 는 normalized preview shell 이었지만, 실제 저장 정본은 shell 안의 raw `.v202-frame-group[data-template-frame-group]` 셀이었다. 그런데 메타데이터와 일부 스타일이 shell 쪽에만 기록되고 정본 셀에는 내려가지 않아, 저장 시 denormalize 이후에는 변경사항이 사라졌다.
   - 해결: `TemplateEditWorkspace.tsx` 에서 메타데이터 읽기/쓰기 기준을 `shell + 실제 persisted frame cell` 양쪽으로 확장했다. `resolvePersistedFrameNode(...)`, `readFrameMetadataAttr(...)` 를 추가해 `boxKind`, `role`, `valueKey`, `parentGroupId`, `runtimeMode` 를 정본 셀까지 함께 읽고 쓰도록 바꿨고, `backgroundColor`, `borderRadius` 도 persisted frame node 에 같이 기록되도록 정리했다.
   - 결과: `선택 항목 저장` 후 `현재 템플릿 저장`, 그리고 새로고침까지 가도 메타데이터와 스타일 변경이 유지된다.
