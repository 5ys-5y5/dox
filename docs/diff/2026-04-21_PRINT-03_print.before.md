# PRINT 설계 문서

## 문서 목적

이 문서는 `/templates/extract` 의 PDF -> HTML 출력 경로를 "출력만 비슷하게 보이는 가짜 복제"가 아니라, **실제로 수정 가능한 HTML 문서 양식과 텍스트**로 재구성하기 위한 최상위 설계 문서다.

이 문서는 특히 아래 두 가지를 동시에 고정한다.

1. 앞으로의 구현은 반드시 `이해 정리 -> 사용자 확정 -> 수정 -> 측정 -> 원인 분석 -> 재수정` 순서를 따른다.
2. `frame` 과 `text` 를 명확히 분리하고, `frame` 에 텍스트 획이 들어가거나 `text` 를 선/점/마스크처럼 출력하는 모든 접근을 금지한다.

이 문서는 `enhance.md`, `buildhtml.md`, `total-todo.md` 보다 우선하는 **print fidelity 전용 실행 기준 문서**다.

## 이번 요청에 대해 확정된 이해

이번 턴에서 확정된 사실은 아래와 같다.

1. 사용자는 `v28` 결과를 실패로 판정했다.
2. 실패 이유는 단순히 점수가 낮아서가 아니라, `frame` 레이어에 텍스트 획처럼 보이는 짧은 선분이 대량으로 들어갔기 때문이다.
3. 사용자가 지향하는 방향은 `v19` 계열의 단순하고 읽기 쉬운 HTML 구현이다.
4. `frame` 은 실제 문서 양식선, 박스, 셀 경계, 체크박스 외곽만 표현해야 한다.
5. `text` 는 실제 텍스트 노드로만 표현해야 한다.
6. 텍스트를 선, 점, 마스크, 비트맵, 도트 벡터, 문자 외곽선 분해 등으로 표현하면 실패다.
7. 시각 유사도는 브라우저 감상이 아니라 **로그에 기록된 실제 측정값**으로 판단해야 한다.
8. 앞으로는 어떤 구현이든 먼저 이해 목록을 정리하고, 사용자가 명시적으로 `확정`한 뒤에만 수정해야 한다.
9. 이번 턴의 목표는 구현이 아니라 위 내용을 강제하는 `docs/print.md` 설계 문서를 만드는 것이다.

## 이번 턴에서 확인한 실패 사실

### 기준 로그

이번 문서가 직접 기준으로 삼는 대표 로그는 아래다.

1. `docs/2026-04-20_23-28-32_28_template-extract-log_9f44fd1b-c0f8-4543-a73a-1b204d18a8a5.md`
2. `docs/2026-04-20_23-11-27_28_template-extract-measurement_terminal-1776694287437.md`

### 로그에서 확정된 실패 패턴

위 로그들로부터 확정된 실패 패턴은 아래와 같다.

1. `cloneBuilder = work_order_family_frame_segment_text_layer_digital`
2. `data-template-layer="frame"` 내부에 수십, 수백 개의 매우 짧은 `horizontal` 선분이 들어갔다.
3. 이 선분들은 실제 양식 프레임선이 아니라 텍스트 획 또는 텍스트 주변 잉크를 프레임으로 오인한 결과다.
4. 즉 `frame extractor` 가 `text ink` 를 `frame ink` 로 잘못 분류했다.
5. 이 상태는 HTML 출력 관점에서도 실패고, 측정 관점에서도 실패다.
6. `frame` 에 text-like segment 가 섞이면 프레임 검토와 텍스트 검토를 분리할 수 없어진다.
7. 따라서 이후 구현은 `frame` 과 `text` 의 도메인 경계를 더 엄격하게 가져가야 한다.

### 이번 실패를 통해 금지로 확정된 행동

1. 텍스트 획을 프레임선으로 출력하는 것
2. 텍스트를 점/마스크/벡터 잉크로 분해해 visible output 으로 쓰는 것
3. 높은 점수만 보고 출력의 수정 가능성과 구조 적합성을 무시하는 것
4. 하나를 고치면서 다른 영역을 동시에 부수는 대규모 혼합 수정
5. 사용자 확정 없이 구현 방향을 임의로 바꾸는 것

## 절대 금지 패턴

아래 항목은 앞으로 최종 출력, 중간 후보, hidden overlay, fallback 어느 위치에도 허용하지 않는다.

1. full-page bitmap background
2. `<img>` 기반 페이지 복제
3. `background-image` 기반 페이지 복제
4. dot cloud
5. point mask
6. glyph outline vector print
7. frame layer 안에 text-like micro segment 출력
8. text 를 `rect`, `line`, `path` 다발로 찍는 방식
9. 텍스트를 수정 가능한 `span/div` 대신 획 출력으로 대체하는 방식
10. `frame` 과 `text` 를 하나의 레이어 안에 섞어서 디버깅 불가능하게 만드는 구조
11. 실제 측정값이 내려갔는데도 “구조적으로 맞아 보인다”는 이유로 유지하는 행위

## 최종 목표

최종 목표는 아래다.

1. `data-template-layer="frame"` 가 실제 양식 프레임만 100% 수준으로 표현한다.
2. `data-template-layer="text"` 가 실제 텍스트만 100% 수준으로 표현한다.
3. 두 레이어를 합친 HTML 출력이 PDF와 `1px 허용 오차 기준 중첩률` 최소 `90%` 이상을 달성한다.
4. 이후 목표는 `95%`, 장기 목표는 `99%`다.

이 문서 기준에서 `90% 미만`은 성공으로 취급하지 않는다.

## 핵심 설계 원칙

### 1. 레이어 도메인 분리

페이지는 반드시 아래 최상위 sibling 구조를 가진다.

1. `data-template-layer="frame"`
2. `data-template-layer="text"`

각 레이어는 아래 역할만 가진다.

#### frame layer

1. 표의 가로선
2. 표의 세로선
3. 외곽 박스
4. 셀 경계
5. 체크박스 외곽
6. 구획선

frame layer 가 가지면 안 되는 것:

1. 글자 획
2. 텍스트 외곽선
3. 서명 필기 잉크
4. 로고 윤곽
5. 도장 텍스처
6. 글자 내부/주변의 짧은 bitmap-like segment

#### text layer

1. 실제 텍스트 문자열
2. 실제 줄 단위 또는 run 단위 배치
3. 편집 가능한 HTML text node
4. placeholder text

text layer 가 가지면 안 되는 것:

1. 선분
2. 점묘
3. 마스크
4. glyph outline path
5. 문자 획을 잘게 쪼갠 rect

### 2. frame 은 "실제 프레임 후보" 에서만 나온다

`frame` 은 아래 우선순위로만 만든다.

1. PDF 벡터/rule segment
2. page geometry 에서 검증된 long-thin cell boundary
3. 검증된 box edge

아래에서 frame 을 만들면 안 된다.

1. text glyph 잉크
2. OCR bitmap 잔상
3. mask-vector 로 검출된 짧은 segment 전체
4. 문자 bbox 가장자리

### 3. text 는 "실제 텍스트 데이터" 에서만 나온다

`text` 는 아래 우선순위로만 만든다.

1. PDFKit line selection
2. PDFKit character bounds
3. run/word grouping
4. OCR line text

아래에서 text 를 만들면 안 된다.

1. frame segment 조합
2. pixel mask
3. glyph outline
4. dot cluster

### 4. 측정 루프를 절대 깨지 않는다

모든 개선은 아래 순서만 허용한다.

1. 현재 기준 로그를 읽는다.
2. 실제 점수가 내려간 원인을 하나로 정의한다.
3. 그 원인만 겨냥한 수정 하나를 넣는다.
4. 터미널에서 초안 생성과 시각 유사도 측정을 다시 실행한다.
5. 점수가 올랐는지, frame/text 도메인이 깨지지 않았는지 둘 다 확인한다.
6. 둘 중 하나라도 실패하면 폐기한다.

## 실행 정책 (필수 준수)

아래 정책은 본 설계 또는 후속 수정에서 100% 준수한다. 간결하게 요약하지 않고, 실제 실행 단계에서 누락이 없도록 상세하게 기록한다.

### 0. 서비스 독립성 설계 원칙 (필수 준수)

아래 기능들은 처음부터 “하나의 서비스로 분리 가능한 단위”로 설계한다.
각 기능은 단순 내부 모듈이 아니라, 향후 별도 배포/운영/API 상품화가 가능해야 한다.

1. 각 기능은 명확한 도메인 경계, 책임, 입력, 출력, 저장소 범위를 가진다.
2. 다른 기능 구현 세부사항(DB 스키마, 내부 함수, 화면 상태)에 직접 의존하지 않고, 오직 계약된 API / 이벤트 / DTO 로만 연결한다.
3. 공통 로직이 필요해도 우선 공통 유틸이 아니라 독립 서비스로 유지 가능한지 먼저 검토한다.
4. 특정 기능이 다른 기능 없이도 테스트, 운영, 교체, 확장 가능하도록 설계한다.
5. 설계 문서에는 반드시 다음을 기능별로 분리해 작성한다.
   - 기능 목적
   - 단독 서비스로서의 가치
   - 책임 범위 / 비책임 범위
   - API 계약
   - 데이터 소유권
   - 의존 서비스
   - 분리 배포 시 필요한 최소 조건
6. 구현안이 나오면 반드시 “이 기능을 지금 당장 별도 서비스로 분리해도 성립하는가?”를 기준으로 검토하고, 성립하지 않으면 결합 지점을 먼저 제거한다.
7. 임시 편의성보다 서비스 경계 보존을 우선한다.

### 1. 모든 코드는 그 어떤 LLM이 보더라도 히스토리와 의도, 목표가 이해될 수 있도록 작성해야 한다.

1. 정의명, 주석 등을 적극 활용한다.
2. 해당 코드를 읽어낸 LLM은 히스토리를 몰라도 구현 상태를 이해할 수 있어야 한다.
3. "임시", "대충", "후에 제거" 수준의 분기 남발을 금지한다.
4. frame 전용 코드, text 전용 코드, measurement 전용 코드는 이름만 보고도 역할이 드러나야 한다.

### 2. 프론트에 UI에 대한 변경이 있는 경우 현재 서비스와 동일한 UI디자이너가 기획한 것과 같은 UI를 구현하여야 한다.

1. `/src/components` 폴더를 수정하지 말고 참고하여 이미 사용된 UI컴포넌트를 최대한 적극적으로 활용한다.
2. `/app` 에 구현된 다른 페이지를 적극적으로 참고하여 구현 상태를 최대한 따라한다.
3. 필요시 최소 필수 파일만 예외로 허용한다.
4. 측정 상태 UI를 추가하더라도 기존 extract 페이지의 디자인 톤을 벗어나지 않는다.

### 3. 수정 전 이해확정 절차

1. 수정 적용 전, 현재 요청에 대한 이해 내용을 목록으로 정리한다.
2. 정리된 이해 내용에 대해 서로 실행하고자 하는 바가 일치하는지 사용자가 명시적으로 확정한 뒤에만 수정한다.
3. 확정 없이 임의로 수정에 착수하지 않는다.
4. 후속 구현 턴에서는 반드시 첫 응답에 아래 3가지를 포함한다.
   - 이번 수정의 목표
   - 건드릴 체크리스트 ID
   - 예상 수정 파일 화이트리스트
5. 사용자가 `확정`하지 않으면 코드 수정, 로그 생성, 버전 추가를 하지 않는다.

### 4. 변경 기록 및 롤백 보장

1. 코드 수정이 있는 경우, 수정 직전의 코드를 반드시 `docs/diff` 폴더에 기록한다.
2. 기록이 없으면 치명적 에러를 막을 수 없으므로, 기록 누락은 허용하지 않는다.
3. 기록 대상은 변경된 파일 전체 또는 수정 구간을 포함하는 형태여야 하며, 언제든 수정 직전 상태로 롤백 가능해야 한다.
4. diff 기록 파일명에는 날짜와 체크리스트 ID를 포함한다.
5. diff 기록 문서에는 어떤 실험이 왜 필요했는지 한 줄 이상 남긴다.

### 5. 확정 범위 외 수정 금지

1. 사용자가 확정한 범위를 넘어서는 변경을 임의로 수행하지 않는다.
2. 서비스 파괴(인코딩, UI)의 주된 원인이므로 절대 금지한다.
3. 새 파일 수정이 필요하면 먼저 화이트리스트 후보로 제안하고 승인받는다.
4. 승인되지 않은 파일은 읽을 수 있어도 수정하지 않는다.

### 6. 체크리스트 작성

1. 설계의 각 항목은 diff에 기록되는 내용의 기준이 되는 지점의 체크리스트를 작성한다.
2. 실제로 어디까지 구현이 완료되었는지 LLM이 기록해나갈 수 있어야 한다.
3. diff 문서에서는 체크리스트 명을 같이 기록하여 어떤 내용의 진행이 어떤 diff문서로 기록되었는지 직접 연관지을 수 있어야 한다.
4. 체크리스트는 "측정", "frame", "text", "composition", "terminal", "ui", "mcp" 수준으로 분리한다.

### 7. MCP 테스트 의무

1. 매 실행마다 `supabase` MCP와 `chrome-devtools` MCP로 의도대로 동작하는지 확인한다.
2. DB 조회로 올바로 등록되었는지 확인이 아닌 DB 수정이 있는 경우 SQL 쿼리를 제공하여 사용자가 직접 실행하도록 한다.
3. 테스트 수행/결과는 문서 하단 체크리스트 및 테스트 기록에 남긴다.
4. MCP 실패 자체도 무시하지 않고 기록한다.
5. MCP가 막혀도 측정과 개선은 터미널 로그 기준으로 이어가되, 막힌 원인을 그대로 남긴다.

## 올바른 측정 기준

앞으로의 모든 판단은 아래 명령으로 생성한 터미널 로그를 기준으로 한다.

```bash
npm run template-extract:measure -- --file docs/작업지시서_사일동\ 주상복합.pdf --version <VERSION> --minimumPassScore 0.95
```

이 명령은 최소한 아래 산출물을 남겨야 한다.

1. `*_template-extract-measurement_*.md`
2. `*_template-extract-log_*.md`

### 측정 해석 규칙

1. `overallScore` 만 주 지표로 사용한다.
2. `sourceInkPixelCount`, `replicaInkPixelCount`, `overlapInkPixelCount` 는 원인 분석용 보조 지표다.
3. `replicaInkPixelCount` 가 커졌는데 `overallScore` 가 떨어지면 overdraw regression 으로 간주한다.
4. `replicaInkPixelCount` 가 너무 작고 `overallScore` 가 낮으면 underdraw regression 으로 간주한다.
5. `frame` 과 `text` 를 분리 렌더한 비교 로그가 가능하면 반드시 같이 남긴다.

## v28 실패에 대한 원인 정의

이번 문서 기준으로 `v28`의 직접 원인은 아래처럼 정의한다.

1. `frame extractor` 가 actual frame line 대신 text-like micro segment 를 대량 수용했다.
2. 그 결과 `frame layer` 가 "문서 양식"이 아니라 "텍스트 획의 잔상"을 출력했다.
3. 이는 `frame` 의 도메인을 깨뜨린 회귀다.
4. 따라서 다음 구현은 `frame` 의 양을 늘리는 것이 아니라 `frame candidate acceptance` 를 엄격히 재정의하는 방향이어야 한다.

### 다음 구현의 강제 방향

1. `frame` 후보는 길이, 두께, 정렬, 반복성, 축 일치, 행/열 경계 일치로 검증한다.
2. 문자 주변의 1~8px 수준 micro segment 는 frame 후보에서 원천 제외한다.
3. `text` 후보는 line/run data 에서만 생성한다.
4. `text` 를 보정한다는 이유로 frame layer 를 건드리지 않는다.
5. `frame` 보정은 `frame` 값만 바꾸고, `text` 보정은 `text` 값만 바꾼다.

## 후속 구현 순서

### 1단계. 이해 확정

후속 구현 턴의 첫 응답은 반드시 아래 형식을 따른다.

1. 이번 수정 목표
2. 건드릴 체크리스트 ID
3. 수정 허용 화이트리스트
4. 예상 기준 로그
5. 사용자 확정 대기

### 2단계. 기준 로그 읽기

1. 직전 best measurement log 를 읽는다.
2. 이번에 고칠 단일 원인을 한 줄로 정의한다.
3. 그 원인이 frame 인지 text 인지 명시한다.

### 3단계. 단일 수정

1. frame 문제면 frame 관련 파일만 수정한다.
2. text 문제면 text 관련 파일만 수정한다.
3. 두 문제를 한 턴에서 동시에 풀지 않는다.

### 4단계. 터미널 측정

1. 동일 PDF
2. 동일 명령
3. 동일 tolerance
4. 새 로그 파일 생성

### 5단계. 유지/폐기

1. 점수 상승 + 도메인 규칙 준수면 유지
2. 점수 하락이면 폐기
3. 점수 상승이 있어도 금지 패턴이면 폐기
