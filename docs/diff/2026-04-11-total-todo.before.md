# 서명 기능 구현

## 전자 서명 기능 구현

- [✓] 무결성 기능
  - 현재 상태: 구현 완료
  - 남은 일: 본인 인증 기능 연결만 남음
  - 코드 흔적: 후속 작업이 필요하다는 내용을 코드와 SQL에 남겨두었음
  - 참고: `pending -> signed` 임시 허용은 본인 인증 실연동 후 제거해야 함
- 본인 인증 기능
  - 현재 상태: 내부 코드와 DB 구조는 준비되어 있음
  - 외부 대기 사유: 사업자 번호 발급 후 BaroCert/PASS 실연동 가능
  - 남은 일: BaroCert, PASS/휴대폰 본인확인 실제 연동 후 `authenticated -> signed`만 허용하도록 축소

# 서류 관리

## 서류 누락 방지

- 서류 클라우드 관리
  - 목표: 모든 서류를 현장별, 종류별, 상태별로 클라우드에서 보관하고 찾을 수 있게 함
- 현장별 필요 서류 누락 방지 기능
  - 목표: 현장 개설 시 공종별 필수 서류 체크리스트를 자동으로 만들어 누락 여부를 확인함

## 반복 서류 자동 생성

- 템플릿 등록
  - 목표: 업로드된 문서를 재사용 가능한 HTML 템플릿과 라벨 구조로 저장함
- 템플릿 추출
  - 목표: 값이 입력된 문서를 읽고 새 템플릿 초안을 자동으로 만듦

## 현장 입력과 승인 처리

- 서류와 연계된 현장별 사진 라벨링 보관
  - 목표: 현장 사진을 라벨과 함께 보관하고, 필요한 라벨 누락을 줄임
- 일괄 정보 입력
  - 목표: 같은 라벨을 가진 여러 문서에 내용을 한 번에 입력, 수정, 삭제함
- 일괄 요청
  - 목표: 특정 라벨만 제한적으로 입력할 수 있는 링크를 메일, 문자로 발송함
- 변환 저장
  - 목표: HTML 정본을 PDF, DOCX, HWP 형식으로 변환하여 저장함

# 기능 설명

* 무결성 기능:
  - 비개발자 설명: 서명하려는 문서가 나중에 몰래 바뀌지 않았는지 확인하는 기능입니다.
  - 현재 구현 범위:
    - 문서 해시 저장
    - 서명 시점 해시 재검사
    - 요청 시점과 서명 시점 문서 일치 검사
    - 감사 로그 기록
    - DB 제약과 트리거로 위변조 방지
    - 검증 SQL로 실제 데이터 확인 가능
  - 현재 코드 위치:
    - `src/lib/crypto.ts`
    - `src/services/signService.ts`
    - `src/app/api/sign/route.ts`
  - 현재 SQL 기록 위치:
    - `docs/applied/run-this-supabase-integrity.sql`
    - `docs/applied/run-this-supabase-integrity-tighten.sql`
    - `docs/applied/run-this-supabase-integrity-signature-guard.sql`
    - `docs/applied/run-this-supabase-integrity-request-guard.sql`
    - `docs/applied/setup-db.sql`
  - LLM 구현 기준:
    - `AUTH_GATE_POST_INTEGRATION_REQUIRED` 흔적을 따라가면 본인 인증 연동 후 마무리해야 할 지점을 찾을 수 있어야 합니다.
    - 무결성 기준 원본은 문서 내용 자체이며, 단순 파일명 비교로 대체하면 안 됩니다.

* 본인 인증 기능:
  - 비개발자 설명: 실제로 서명하는 사람이 본인인지 확인한 뒤에만 서명되게 만드는 기능입니다.
  - 현재 상태:
    - 내부 구현은 준비됨
    - 외부 실연동은 사업자 번호 발급 후 진행
  - 현재 이미 구현된 내부 범위:
    - `AUTH_REQUEST`
    - `AUTH_STATUS`
    - `AUTH_CANCEL`
    - `AUTH_VERIFY`
    - `AUTH_CALLBACK`
    - `sign_authentications` 기록 구조
    - `authenticated` 상태에서 `verified` 인증 기록 검사
    - `consent_text_hash` 일치 검사
    - `ci/di`, `signed_data`, raw response 해시 저장
  - 현재 코드 위치:
    - `src/lib/authProviders.ts`
    - `src/lib/authEvidence.ts`
    - `src/services/signAuthService.ts`
    - `src/services/signService.ts`
    - `src/app/api/sign/route.ts`
    - `src/app/test-sign/page.tsx`
  - 현재 SQL 기록 위치:
    - `docs/applied/run-this-supabase-auth-bootstrap.sql`
    - `docs/applied/setup-db.sql`
  - 외부 연동 후 마지막으로 해야 할 일:
    - BaroCert 토스 실제 요청 연결
    - BaroCert 카카오, 네이버 연결
    - NICE/PASS callback 연결
    - 운영용 키 반영
    - `pending -> signed` 제거
    - `authenticated -> signed`만 허용
  - LLM 구현 기준:
    - 인증 성공 후 `sign_requests.status = authenticated`가 되어야 합니다.
    - 서명 실행 시 `sign_authentications.auth_status = verified`가 있어야 합니다.
    - 인증 시점 `document_hash`, `consent_text_hash`와 서명 시점 값이 같아야 합니다.

* 서류 클라우드 관리:
  - 비개발자 설명: 모든 서류를 현장별, 종류별, 상태별로 클라우드에서 보관하고 찾을 수 있게 하는 기능입니다.
  - 목표:
    - 현장별 서류 목록 확인
    - 최신본과 이전본 구분
    - 상태별 검색
    - HTML 정본과 변환본 분리 보관
  - LLM 구현 기준:
    - 문서는 파일만 저장하지 말고 `문서 메타데이터 + HTML 정본 + 출력본`으로 분리해야 합니다.
    - 삭제는 가능하면 바로 완전 삭제보다 soft delete를 우선 검토해야 합니다.

* 현장별 필요 서류 누락 방지 기능:
  - 비개발자 설명: 새 프로젝트가 열릴 때마다 어떤 서류가 필요한지 다시 판단하다가 누락하는 일이 없도록, 현장 개설 시 미리 입력된 공종별 필수 서류 체크리스트에 맞게 서류를 챙겼는지 확인할 수 있는 기능입니다.
  - 목표:
    - 현장 생성 시 자동 체크리스트 생성
    - 누락 서류 표시
    - 작성중, 완료 상태 구분
    - 공종별 필수 서류 규칙 재사용
  - LLM 구현 기준:
    - 체크리스트는 파일이 아니라 규칙 테이블로 관리해야 합니다.
    - 한 현장에 여러 공종이 있으면 필요한 서류를 합쳐서 계산해야 합니다.
    - 같은 서류가 여러 공종에서 겹치면 한 항목으로 합쳐야 합니다.

* 템플릿 등록:
  - 비개발자 설명: 업로드 된 템플릿을 시스템이 다시 쓸 수 있는 전자 양식으로 바꾸는 기능입니다.
  - 필수 동작:
    - 업로드 된 템플릿은 LLM이 텍스트와 문서 틀을 그대로 HTML로 그려내어 관리해야 합니다.
    - Google Docs에서 표를 입력하는 것처럼, 입력된 내용에 따라 문서 틀의 높낮이나 너비가 달라질 수 있어야 합니다.
    - 높이를 늘릴지, 너비를 늘릴지는 사용자가 선택할 수 있어야 합니다.
  - 라벨링 규칙:
    - 빈 템플릿 문서를 업로드하면 입력 항목을 미리 라벨링할 수 있어야 합니다.
    - 다른 문서라도 같은 라벨이면 관련 정보를 일괄 수정할 수 있어야 합니다.
    - 라벨링 항목에는 서명도 포함됩니다.
    - 서명은 라벨과 함께 영역을 지정하면 그 영역이 해당 라벨의 서명 영역이 되어야 합니다.
    - 서류가 달라도 같은 라벨은 같은 서명이 입력될 수 있어야 합니다.
  - 서명 묶음 규칙:
    - 예를 들어 안전 책임처럼 같은 담당자가 여러 문서에 일괄 서명해야 한다면 해당 담당자가 서명할 수 있는 문서 리스트가 보이고, 선택한 문서는 한 번에 정보와 요약을 확인한 뒤 서명할 수 있어야 합니다.
  - LLM 구현 기준:
    - 템플릿 등록 결과는 최소 `HTML 레이아웃`, `필드 스키마`, `라벨 매핑`이 같이 저장되어야 합니다.
    - 필드 스키마에는 최소 `field_id`, `label_key`, `field_type`, `page`, `bbox`, `layout_resize_mode`가 필요합니다.
    - `layout_resize_mode`는 최소 `fixed`, `grow_height`, `grow_width`를 구분해야 합니다.

* 템플릿 추출:
  - 비개발자 설명: 값이 입력된 항목은 LLM이 문서를 읽고 템플릿을 만들어낼 수 있어야 합니다.
  - 목표:
    - 고정 문구와 입력값 구분
    - 표 구조 유지
    - 기존 라벨과의 유사도 비교
    - 사용자 검토 후 정식 템플릿 저장
  - LLM 구현 기준:
    - 바로 확정 저장하지 말고 초안 상태를 먼저 만들어야 합니다.
    - 확신이 낮은 필드는 `검토 필요` 상태를 남겨야 합니다.

* 서류와 연계된 현장별 사진 라벨링 보관:
  - 비개발자 설명: 각 현장별 사진은 현장 ID와 라벨을 통해서 관리되며, LLM이 사진을 보고 어떤 라벨이 붙어야 하는지 제안해서 필요 라벨을 누락하는 일이 없도록 해야 합니다.
  - 목표:
    - 현장별 사진 업로드
    - 사진별 여러 라벨 지정
    - LLM 라벨 추천
    - 문서가 요구하는 사진 라벨 누락 경고
  - LLM 구현 기준:
    - 사진은 반드시 `site_id`와 연결되어야 합니다.
    - 추천 라벨은 자동 확정하지 말고 사용자 검토 단계를 둬야 합니다.

* 일괄 정보 입력:
  - 비개발자 설명: 동일한 라벨을 가지고 있는 경우 선택한 문서는 내용을 일괄 입력, 수정, 삭제가 가능해야 합니다. 서명도 포함됩니다.
  - 목표:
    - 여러 문서 선택
    - 여러 라벨 선택
    - 입력, 수정, 삭제 선택
    - 적용 전 미리보기
    - 작업 이력 저장
  - LLM 구현 기준:
    - 타입이 같은 라벨끼리만 일괄 처리해야 합니다.
    - 서명 라벨은 일반 텍스트와 다르게 추가 권한 검사와 본인확인 검사를 붙여야 합니다.

* 일괄 요청:
  - 비개발자 설명: 문서의 라벨을 선택하여 메일, 문자를 입력하면 해당 메일과 문자로 그 부분만 제한적으로 입력할 수 있도록 문서 링크를 보낼 수 있어야 합니다.
  - 추가 설명:
    - 템플릿을 HTML로 구현한 것처럼, 문서 작성과 수정도 웹 페이지에서 가능해야 합니다.
    - 수신자는 허용된 부분만 수정할 수 있어야 합니다.
  - 목표:
    - 제한 라벨 선택
    - 수신자 지정
    - 만료일 설정
    - 링크 발송
    - 링크 토큰 검증
    - 수정 기록 남기기
  - LLM 구현 기준:
    - 요청 링크는 토큰 기반이어야 합니다.
    - 허용된 라벨 외에는 수정이 불가능해야 합니다.
    - 누가 언제 무엇을 바꿨는지 기록해야 합니다.

* 변환 저장:
  - 비개발자 설명: 템플릿과 문서는 내부적으로 HTML로 구현되지만, PDF, Word 문서(`.docx`), HWP 양식으로 수정 가능하게 저장할 수 있어야 합니다.
  - 목표:
    - HTML -> PDF
    - HTML -> DOCX
    - HTML -> HWP
    - 어떤 버전에서 만든 출력물인지 기록
  - 중요한 원칙:
    - 내부 정본은 HTML입니다.
    - PDF는 보기와 보관용입니다.
    - DOCX와 HWP는 수정 가능한 배포본입니다.
    - 외부에서 수정한 DOCX나 HWP는 자동으로 내부 정본이 되면 안 됩니다.
  - LLM 구현 기준:
    - 변환은 job 단위 비동기 처리로 설계하는 편이 안전합니다.
    - HWP는 별도 변환기나 외부 엔진이 필요할 가능성이 높습니다.
    - 정본 원본과 변환 결과물은 분리 저장해야 합니다.
